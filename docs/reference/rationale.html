Internally, **EVE**'s implementation uses a simplified [Niebloid][niebloid] design to achieve such goals. This choice is driven by multiple reasons: - It makes all **EVE** "functions" directly usable as an argument to another function. Contrary to template functions, function objects cna be easily passed as parameters to algorithms or other higher-order functions. - It enables the [definition of function decorators](#functioncustomizationviahigher-orderdecorator) that acts as lightweight wrapper for modifying the behavior of a function at compile-time. - It allow for a simplified API for [masked operations](#maskedoperationssupport) as the logical mask can be applied directly on the function object. Those so-called **Masked operations** are accessible in **EVE** via the following protocol: - Let `func` be an **EVE** object function object that supports masks and let `cond` be a :ref:`Logical Value`. - Then, for any [value](../concepts.html#value) `x`, the expression `func[cond](x, ...)` is equivalent to `if_else(cond, func_(x,...), x)`. - And, for any [value](../concepts.html#value) `x`, the expression `func.not_[cond](x,...)` is equivalent to `if_else(!cond, func_(x,...), x)`. Note that the negation of the condition is not evaluated. Basically, if the mask is set to `false` for a given lane, the value returned by the function is the corresponding value of `x`, irregardless of other parameters. Note that this semantic implies that `x` must be convertible to `decltype(func(x))`. The availability of such syntax is indicated explicitly in each function documentation page. ----------------------------------------------------------------------------------------------------
